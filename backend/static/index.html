System Calculation Logic

CPU Usage

ΔTotal − ΔIdle ΔTotal × 100

ΔTotal: Change in total CPU time between two readings from /proc/stat.

ΔIdle: Change in idle CPU time (when CPU is not executing tasks).

CPU usage is calculated by comparing how much time the CPU was actively working versus how much time it remained idle over a short interval.

Memory Usage

MemTotal − MemAvailable MemTotal × 100

MemTotal: Total physical memory available in the system (from /proc/meminfo).

MemAvailable: Memory that can be used by new processes without swapping.

Memory usage represents the portion of total memory currently in use, considering usable memory instead of just free memory.

Load Generation & Graph Behavior

The spikes observed in CPU and Memory graphs are generated using controlled load mechanisms implemented in the system.

CPU Load Generation:

CPU load is generated by spawning multiple background processes (using yes command). These processes continuously consume CPU cycles, increasing CPU utilization and causing visible spikes in the graph.

Memory Load Generation:

Memory load is created by allocating blocks of memory dynamically in the backend using byte arrays. As more memory blocks are allocated, memory usage increases, which is reflected as spikes in the graph.

Graph Behavior:

The frontend continuously fetches system metrics at regular intervals. Any increase or decrease in resource usage due to load generation is immediately reflected as spikes or drops in the graphs.

Process Creation & Execution Logic

Processes used in the CPU scheduling simulation are dynamically created based on user input. Each process is defined by its Arrival Time (AT) and Burst Time (BT).

Process Creation:

When the user adds processes in the scheduler panel, the frontend collects the input values and sends them to the backend via an API request. The backend then organizes these processes according to the selected scheduling algorithm (currently FCFS).

Execution Mechanism:

The backend simulates execution by processing each task sequentially. Each process runs for its specified burst time using timed delays, representing actual CPU execution duration.

Impact on CPU Usage:

During execution, artificial CPU load is generated by spawning background processes. These processes continuously consume CPU cycles, causing an increase in CPU utilization which is reflected as spikes in the CPU usage graph.

Impact on Memory Usage:

Memory usage increases as the system allocates memory blocks dynamically during execution. Each process contributes to memory consumption, and the cumulative allocation results in visible changes in memory usage on the graph.

Real-Time Synchronization:

The frontend continuously fetches execution status and system metrics from the backend. This ensures that process execution, CPU load, and memory usage are visualized in real-time.
